# AI 错误模式库

> 🤖 **AI 执行指令：每次会话开始时，快速扫描本文件中所有"预防规则"条目。当前任务场景匹配到任一错误模式的关键词时，必须在动手前复述对应的预防规则。每次犯新错误时，按下方模板追加一条记录。**

---

## 输入源

| 输入源 | 触发时机 | 同步规则 |
|--------|---------|---------|
| 会话直接发现 | AI 犯错 → 用户指出 → 当场记录 | 按模板直接追加 |
| gstack learnings | /learn 命令或健康扫描时 | 比对后分配 E 编号 |

---

## 使用方式

### 对人
碰到 AI 反复犯同一类错误 → 来这里登记一条 → 下次任何 AI 工具读到都会预防。

### 对 AI
1. 会话开始：扫一遍场景关键词，匹配当前任务则复述预防规则
2. 犯错当下：立刻按模板追加一条
3. 写代码/做决策前：任务关键词命中某条 → 先复述预防规则再动手
4. 执行 /learn 后：比对 gstack learnings 与本库，缺失则分配编号追加

## 错误记录

### E001：中文正则词边界失效

| 字段 | 内容 |
|------|------|
| 日期 | 2026-07-02 |
| 来源 | 会话直接发现 |
| 场景关键词 | `正则`, `regex`, `中文`, `CJK`, `关键词匹配`, `文本搜索`, `Python` |
| 现象 | 用正则匹配中文文本中的关键词，结果全为假阴性 |
| 根因 | Python 正则引擎不认 CJK 字符，词边界对中文完全无效 |
| 预防规则 | **处理中文文本时，禁止使用词边界。改用简单子串匹配 `keyword in text` 或 `text.find(keyword)`。** |

### E002：宽泛关键词导致假阳性泛滥

| 字段 | 内容 |
|------|------|
| 日期 | 2026-07-02 |
| 来源 | 会话直接发现 |
| 场景关键词 | `关键词`, `分类`, `匹配`, `过滤`, `数据清洗`, `假阳性` |
| 现象 | 用泛词做关键词匹配，大量不相关条目被错误归类 |
| 根因 | 单个泛词命中范围太宽，没有用复合短语限定语义 |
| 预防规则 | **关键词匹配必须用具体复合短语，禁止使用单字泛词。新增关键词前先验证：这个词会不会误匹配不相关文本？** |

### E003：表格文件列结构不一致

| 字段 | 内容 |
|------|------|
| 日期 | 2026-07-02 |
| 来源 | 会话直接发现 |
| 场景关键词 | `表格`, `解析`, `列偏移`, `Excel`, `多Sheet` |
| 现象 | 不同 Sheet 的同一字段在不同列，按列号硬编码导致数据丢失 |
| 根因 | 不同 Sheet 列结构不一致，按列号而非列名匹配 |
| 预防规则 | **解析多 Sheet 表格：先读表头确认列名，按列名匹配而非按列号，关键列加兜底逻辑。** |

### E004：触发条件写太窄导致规则不生效

| 字段 | 内容 |
|------|------|
| 日期 | 2026-07-02 |
| 来源 | 会话直接发现 |
| 场景关键词 | `触发条件`, `规则`, `指令`, `AI执行`, `首次部署`, `入口文件` |
| 现象 | 规则触发条件只检查文件是否存在。空文件/占位符被当作已部署，后续追问链失效 |
| 根因 | 检查条件是文件是否存在而非文件内容是否完整 |
| 预防规则 | **写 AI 执行指令的触发条件时，检查内容是否满足要求而非文件是否存在。文件存在 → 有目标段落 → 内容非模板占位符，任一步不满足则触发追问。** |

### E005：Markdown 表格转义导致断链误报

| 字段 | 内容 |
|------|------|
| 日期 | 2026-07-02 |
| 来源 | 会话直接发现 |
| 场景关键词 | `断链检查`, `wikilink`, `markdown表格`, `正则`, `false positive`, `Obsidian` |
| 现象 | 正则提取 wikilink 时，表格中的别名链接被解析为带反斜杠的路径，已有文件被误报为断链 |
| 根因 | markdown 表格中转义字符未被清理 |
| 预防规则 | **断链检查中 wikilink target 提取后必须清理转义字符。流程：正则匹配 → 去别名 → 去转义 → 与文件列表比对。** |

### E006：断链检查未过滤代码块导致 false positive

| 字段 | 内容 |
|------|------|
| 日期 | 2026-07-04 |
| 来源 | gstack learning `obsidian-broken-link-code-block` |
| 场景关键词 | `断链检查`, `wikilink`, `代码块`, `code block`, `正则`, `false positive`, `Obsidian` |
| 现象 | 全量断链扫描报告的断链中，大部分是代码示例中的 `[[...]]`、`` `...` `` 被正则误抓 |
| 根因 | 正则不区分真实 wikilink 和代码块/内联代码中的示例文本 |
| 预防规则 | **断链检查必须分两步：① 用正则剔除所有代码块（\`\`\`...\`\`\`）和内联代码（\`...\`）→ ② 在剩余纯文本中提取 wikilink。不能用同一个正则同时匹配。** |

### E007：Inbox 去重仅按 ID → 同论文多分类重复存储

| 字段 | 内容 |
|------|------|
| 日期 | 2026-07-04 |
| 来源 | gstack learning `inbox-dedup-id-only` |
| 场景关键词 | `Inbox`, `去重`, `dedup`, `arxiv`, `知识收集`, `数据质量` |
| 现象 | 同一篇 arxiv 论文在多个分类下共享相同 ID，去重只按 ID 判断，导致同一论文存多次 |
| 根因 | ID 生成不含分类字段，同论文不同分类共享 ID。去重只靠 ID 检查，标题相似度兜底缺失 |
| 预防规则 | **去重必须双因子检查：① ID 去重（快速路径）→ ② 标题相似度去重（兜底）。相似度 > 80% 视为重复。ID 相同时第一个分类写入，其他追加到 tags 数组。** |

### E008：Inbox 相关性评分形同虚设

| 字段 | 内容 |
|------|------|
| 日期 | 2026-07-04 |
| 来源 | gstack learning `inbox-relevance-semantic-filter` |
| 场景关键词 | `Inbox`, `相关性`, `relevance`, `过滤`, `噪音`, `知识收集` |
| 现象 | Inbox 中标"高"相关的条目包含零相关的生活/娱乐内容。采集器对高分帖自动标"高"，忽略语义 |
| 根因 | 相关性仅根据外部得分判断，没有语义过滤层 |
| 预防规则 | **Inbox 采集必须加语义过滤层：① 标题不含 AI/开发/法律/隐私/安全关键词 → 自动降为"低"或直接丢弃 → ② 仅核心领域相关且高分才标"高"→ ③ 定期审查"高"条目中实际无关的比例。** |

### E009：config_loader 默认 workspace 写错 → 调度系统分裂

| 字段 | 内容 |
|------|------|
| 日期 | 2026-07-04 |
| 场景关键词 | `config`, `workspace`, `路径`, `默认值`, `多AI调度`, `loop_dispatch`, `配置漂移` |
| 现象 | 默认 workspace 少写了 `Pet_`，环境变量未设时回退到错误目录，两个 worker 各读各的任务目录 |
| 根因 | 硬编码默认值与实际目录名不一致，配置加载器不做路径存在性验证 |
| 预防规则 | **所有配置文件的默认路径必须是真实存在的目录。写默认值前 `ls` 确认。配置加载器必须在返回前验证路径存在且可写，否则立即报错。smoke test 必须含默认路径匹配检查。** |

### E010：monitor state JSON 缺键 → KeyError 崩溃

| 字段 | 内容 |
|------|------|
| 日期 | 2026-07-05 |
| 场景关键词 | `KeyError`, `JSON`, `state`, `monitor`, `notified_status`, `向后兼容`, `schema migration` |
| 现象 | `loop_monitor.py check` 在访问 `state["notified_status"]` 时抛出 KeyError，监控完全不可用 |
| 根因 | 代码假定 key 一定存在，但 state 文件是旧版 schema 不含此键，升级代码后未做 migration |
| 预防规则 | **任何从 JSON 文件读取的 dict 访问嵌套键前，必须做防御性初始化。所有 JSON state 文件必须定义 schema version 字段，load() 时检查版本并执行 migration。smoke test 应覆盖空/旧版 state 文件场景。** |

### E011：Vault 操作后只更新当前状态 → 漏写知识库日志和会话沉淀

| 字段 | 内容 |
|------|------|
| 日期 | 2026-07-06 |
| 场景关键词 | `vault`, `修复`, `断链`, `健康检查`, `知识库维护`, `收尾`, `日志`, `会话沉淀`, `跨AI` |
| 现象 | 完成全量 vault 修复（76 断链→0、frontmatter 补全、index 重构），只更新了当前状态.md 和 index.md，漏掉了知识库日志追加和会话沉淀创建。用户追问才发现遗漏 |
| 根因 | AI 只记住了知识库规则中「更新当前状态」这一步，没有完整执行全部收尾步骤。收尾三步的规则分散在知识库规则、自进化机制、AI 优先设计原则三个不同文件中，没有合并成硬触发器 |
| 预防规则 | **任何 vault 修改操作（修复断链、新增文件、重构分类、批量改 frontmatter 等），必须执行完整的「收尾三步」：① 更新 `系统/当前状态.md`（做了什么+决定+待办）→ ② 追加 `日志/知识库日志`（时间戳+分类+操作摘要）→ ③ 判断是否有方法论价值：有则创建 `日志/会话沉淀/YYYY-MM-DD 主题.md`（面向其他 AI 的完整操作文档），无则跳过。三步全部完成才算结束。禁止「改了当前状态就完事」。** |
